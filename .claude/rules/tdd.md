## 프로젝트별 규칙 우선 조항

CLAUDE.md 또는 AGENTS.md에 TDD 관련 규칙이 명시되어 있으면 그 규칙이 **항상 우선**한다.
이 파일은 fallback(기본값)이다.

---

## 적용 기준

### TDD를 반드시 적용해야 하는 대상

- **비즈니스 로직** — 컴포넌트 생성 파이프라인, 프로바이더 선택 로직, 코드 변환 처리
- **API 엔드포인트** — `server/index.ts`의 라우팅, 에러 핸들링(503/429/MAX_TOKENS)
- **유틸 함수** — ID 생성, 텍스트 파싱, 데이터 변환
- **버그 수정** — 수정 전 실패하는 테스트를 먼저 작성해 재현, 이후 수정

### TDD가 불필요한 대상

- 타입 정의 (`src/types/index.ts`)
- 설정 파일 (`vite.config.ts`, `.eslintrc`, `tsconfig.json` 등)
- 순수 UI 컴포넌트 (렌더링 외 로직 없음)
- SQL / 스키마 마이그레이션

## RED → GREEN → REFACTOR 사이클

### RED

- **하나의 동작 = 하나의 테스트.** 여러 동작을 한 테스트에 묶지 마라.
- 테스트를 작성한 직후 **반드시 실행해 실패를 확인하라.**
- 실패 이유는 반드시 **"기능이 아직 구현되지 않음"** 이어야 한다.
  - 컴파일 에러, import 오류, 타입 오류는 RED가 아니다 — 테스트 자체를 고쳐라.

### GREEN

- 테스트를 통과시키는 **최소한의 코드만** 작성한다.
- **YAGNI** — 지금 필요하지 않은 기능은 작성하지 마라.
- 신규 테스트와 기존 테스트 **모두 통과**해야 다음 단계로 넘어간다.

### REFACTOR

- **중복 제거, 이름 개선, 헬퍼 추출**만 허용된다.
- 모든 테스트가 green 상태를 **계속 유지**해야 한다.
- 새로운 동작을 추가하지 마라 — 추가하고 싶다면 다음 RED 사이클에서 시작하라.

### 반복

REFACTOR가 끝나면 **다음 동작에 대한 RED**로 돌아간다.

## 삭제 강제 규칙

- 테스트 없이 프로덕션 코드를 먼저 작성했다면 **즉시 삭제하고 RED부터 재시작**한다.
- "나중에 테스트 추가할 참고용"으로 남기는 것도 **금지**한다.
- 예외 없음.

## 변명 차단표

| 변명 | 반론 |
|------|------|
| "너무 단순해서 테스트 불필요" | 단순한 코드도 회귀한다. 작성 비용은 30초, 디버깅 비용은 30분이다. |
| "나중에 추가하겠다" | 나중은 오지 않는다. 지금 작성하지 않으면 영원히 없다. |
| "시간이 없다" | TDD 없이 작성한 코드의 디버깅 시간이 더 길다. 시간을 버리는 것은 테스트가 아니라 이 변명이다. |
| "삭제하면 낭비" | 테스트 없는 코드는 동작을 보장할 수 없다. 보장 안 되는 코드가 낭비다. |
| "프로토타입이다" | 프로토타입은 버려진다. 버려지지 않으면 그 순간부터 프로덕션이다. |
