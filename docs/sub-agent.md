# code-evaluator Sub-agent 생성 요청(프로젝트 스코프로 생성)

## 강의 맥락

이 실습은 **Part 4 · Hook + Sub-agent**의 마무리 단계다. 3단 구조의 마지막 층을 완성한다.

| 층 | 역할 | 예시 (이전 단계) |
|------|------|------|
| Protection | 위험 작업 사전 차단 | PreToolUse hook |
| Reminder | 규칙 사후 상기 | PostToolUse TDD 리마인더 hook |
| **Sub-agent** | **독립 검증** | **이번 단계 — code-evaluator** |

이 세 층이 합쳐지면 Claude Code는 **단일 도구가 아닌 자기 규율 시스템**이 된다.

## 서브에이전트의 역할 (핵심 개념)

**code-evaluator는 전체 코드베이스를 리뷰하는 에이전트가 아니다.**

- **입력**: 메인 Claude가 **방금 작업한 내용**(특정 파일/변경)
- **작업**: 그 변경에 대해서만 독립 시각으로 점검
- **출력**: 요약 피드백만 메인에 반환
- **환경**: 서브에이전트 **고유 컨텍스트** — 메인 컨텍스트를 오염시키지 않음

즉, 서브에이전트는 **범위가 좁고(scope)**, **컨텍스트가 격리되며(isolation)**, **시각이 독립적인(independence)** 세 가지 속성을 모두 가진다. 하나라도 빠지면 그냥 "작은 Claude 하나 더"에 불과하다.

## 모델 배치: 작업자 ≤ 검증자

**핵심 원칙**: 검증자는 작업자보다 약하면 안 된다.

### 왜 검증자가 더 강해야 하는가

- 약한 검증자는 강한 작업자의 미묘한 실수를 놓친다 (Generator-Verifier Gap)
- "내 코드는 내가 봐도 괜찮아 보인다" 편향은 검증자가 자기 자신이기 때문에 발생 — **서브 분리만으로는 부족하고, 시각도 더 엄격해야** 편향이 실제로 제거된다
- 특히 **숨은 버그, 경계값, 비동기 레이스, 보안 이슈**는 의미 기반 추론이 필요 — 작은 모델이 놓치기 쉬움

### 강의 환경 제약과 역할 다운시프트

강의장에서는 **Opus 사용 불가** (Haiku, Sonnet만 허용). 따라서 역할을 한 단계씩 내려서 구성한다.

| 역할 | 일반 환경 | 강의 환경 |
|------|-----------|-----------|
| 작업자 (메인) | Sonnet | **Haiku** |
| 검증자 (서브) | Opus | **Sonnet** |

원칙은 같다:
- 작업자 ≤ 검증자 (티어 관점에서)
- 검증자는 **읽기 전용 + 좁은 책임 + 더 엄격한 시각**
- 생성과 검증의 분리가 핵심이며, 모델 선택은 그 원칙을 구현하는 도구일 뿐

## 목표

`.claude/agents/code-evaluator.md` 서브에이전트를 생성한다. 메인 Claude가 방금 작업한 코드를 넘겨주면, 이 서브에이전트가 **작업 범위 내에서** 독립 평가하여 요약 피드백을 반환한다.

## 전제 조건

- Claude Code 설치 및 `/agents` 기능 사용 가능 환경
- 이전 단계에서 생성된 `~/.claude/rules/tdd.md` 존재 (검증 체크리스트의 근거로 참조)

## 생성할 파일

`.claude/agents/code-evaluator.md` — 서브에이전트 정의 파일 (신규 생성)

## 파일 구조

### 1. YAML Frontmatter

```yaml
---
name: code-evaluator
description: 메인 Claude가 방금 작업한 코드를 읽기 전용으로 독립 검증한다. TDD 준수, 테스트 커버리지, 네이밍, 단일 책임, 숨은 버그 가능성을 점검하고 요약 피드백만 반환한다. 전체 코드베이스 리뷰가 아닌 작업 범위 한정 검증 전용. 코드를 수정하지 않는다.
tools: Read, Glob, Grep
model: sonnet
---
```

**각 필드의 이유:**

| 필드 | 값 | 이유 |
|------|------|------|
| `tools` | `Read, Glob, Grep` | 읽기 전용 — 검증자가 코드를 수정하면 역할 분리가 깨진다 |
| `model` | `sonnet` | 검증자는 작업자(Haiku)보다 강한 티어. 의미 기반 판단 필요 |
| `description` | 구체적으로 명시 | 메인 Claude가 **언제 이 서브를 호출할지** 자동 판단할 근거. "작업 범위 한정"을 강조 |

### 2. 시스템 프롬프트 본문

Frontmatter 아래에 서브에이전트의 행동 규약을 작성한다. 포함할 섹션:

#### (A) 역할 선언
```
너는 작성자가 아닌 검증자다. 코드를 고치려 하지 말고, 무엇이 문제인지 짚어라.
메인 Claude가 방금 작업한 코드에 대해서만 읽기 전용으로 평가한다.
전체 코드베이스를 리뷰하지 마라 — 메인이 지정한 파일/변경 범위에 집중하라.
작성한 당사자(메인)보다 엄격한 시각으로 본다. 관대한 통과는 검증 실패와 같다.
```

#### (B) 평가 체크리스트 (고정)

1. **TDD 준수** (`~/.claude/rules/tdd.md` 기준)
   - `src/*.ts(x)` 에 대응되는 `*.test.ts(x)` 가 있는가
   - 테스트가 **동작**을 검증하는가, **구현**을 복제하는가
   - RED를 건너뛴 흔적(테스트 없이 추가된 로직)이 있는가
2. **단일 책임**: 함수/모듈이 하나의 이유로만 변경되는가
3. **네이밍**: 의도가 이름에 드러나는가 (주석 없이 읽혀지는가)
4. **숨은 버그** (의미 기반 점검 — 검증자가 더 강해야 하는 이유)
   - null/undefined 처리
   - 경계값 (빈 배열, 0, 음수, 최대값)
   - 비동기 레이스 컨디션, 에러 전파
   - 예외 삼키기, 잘못된 기본값
5. **프로젝트 규칙 준수**: `CLAUDE.md` / `AGENTS.md` 가 있으면 그 규칙 위반 여부

#### (C) 출력 형식 (고정)

메인 컨텍스트 오염을 막기 위해 **요약만** 반환:

```
## 평가 결과

### ✅ 잘된 점 (최대 3개)
- ...

### ⚠ 개선 필요 (우선순위 순, 최대 5개)
1. [심각도: HIGH/MED/LOW] {한 줄 요약}
   - 근거: {파일:라인}
   - 제안: {한 줄}

### 🎯 다음 액션 추천
- {메인 Claude가 바로 실행할 수 있는 구체 지시 1-2개}
```

#### (D) 금지 사항

- 코드 스니펫을 길게 인용하지 마라 (파일:라인만 언급)
- `Edit`/`Write`/`Bash` 를 사용하지 마라 (도구 제한으로 차단되지만 재확인)
- 메인이 지정하지 않은 파일을 광범위하게 탐색하지 마라 — **작업 범위 한정**
- 불확실한 지적은 "추정"임을 명시하라
- 메인의 판단을 대신 내리지 마라 — 근거와 선택지를 제공할 뿐

## 메인 Claude의 호출 방법

서브에이전트는 Claude Code의 `Task` 도구(또는 `/agents` 워크플로)로 호출된다.

**실습 시나리오:**

```
사용자: "src/user-service.ts 에 로그인 로직 추가해줘"

→ 메인 Claude (Haiku): 로직 작성 + 테스트 작성

사용자: "code-evaluator로 검증해줘"
 (또는 메인이 스스로 "작성 완료했으니 서브로 검증해볼까요?" 제안)

→ 메인이 Task 도구로 code-evaluator 서브를 호출
  · 서브 고유 컨텍스트에서 실행 (메인 컨텍스트 격리)
  · 서브가 Read/Glob/Grep으로 해당 파일 + 관련 테스트만 탐색
  · ## 평가 결과 형식의 요약 피드백만 메인에 반환

→ 메인이 피드백을 보고 수정 수행 (또는 사용자에게 확인 요청)
```

**중요**: 메인이 서브를 호출할 때 **작업 범위를 명시**해야 한다. "프로젝트 전체 검증"이 아닌 "방금 수정한 파일 X, Y 검증"으로 호출해야 서브가 올바르게 범위를 좁힌다.

## 검증 절차 (구현 후 필수)

1. **파일 구조 확인**:
   ```bash
   test -f .claude/agents/code-evaluator.md && echo "OK"
   head -6 .claude/agents/code-evaluator.md  # frontmatter 확인
   ```

2. **서브에이전트 등록 확인**: Claude Code에서 `/agents` 명령으로 `code-evaluator` 가 목록에 나타나는지 확인

3. **동작 확인** (실습 시나리오):
   - `src/sample.ts` 에 **의도적으로** 네이밍이 나쁘고 null 처리가 빠진 함수 작성 (테스트 없이)
   - 메인 Claude에게 "code-evaluator로 이 파일만 검증해줘" 요청
   - 확인 사항:
     - 서브가 `## 평가 결과` 형식으로 피드백을 반환하는가
     - **작업 범위 내**의 문제만 지적하는가 (엉뚱한 파일을 뒤지지 않는가)
     - null 처리 누락 같은 **의미 기반 이슈**를 잡아내는가 (약한 모델이 놓칠 것)
     - 서브가 `Edit`/`Write` 를 시도하지 않았는가 (도구 사용 로그 확인)

## 강의에서 강조할 포인트

### 1. Sub-agent는 "작은 Claude 하나 더"가 아니다
세 가지 속성이 동시에 있어야 한다:
- **Scope (범위 좁음)**: 작업 범위 한정, 전체 리뷰 아님
- **Isolation (컨텍스트 격리)**: 메인 컨텍스트를 오염시키지 않고 요약만 반환
- **Independence (시각 독립)**: 작업자 ≤ 검증자 모델 티어

### 2. 검증자를 약하게 두지 마라
일반적인 직관("검증은 쉬우니 작은 모델로")은 **규칙 기반 체크**에만 유효하다.
- 체크리스트(테스트 유무, 린트) → 작은 모델 OK
- **의미 기반(숨은 버그, 로직 오류)** → 검증자가 작업자보다 크거나 같아야 함
- 이 강의에서는 의미 기반 검증까지 다루므로 Sonnet 검증자를 선택

### 3. 자기 평가 편향 제거
"내 코드는 내가 봐도 괜찮아 보인다" — 이는 검증자가 **작성자 자신**이기 때문에 발생.
Sub-agent 분리는 **단순한 프로세스 분리가 아니라 편향 제거 장치**다.

### 4. 3단 구조의 완성
- **Protection (PreToolUse 차단)**: 사고가 일어나기 전에 막는다
- **Reminder (PostToolUse 경고)**: 일어난 후 부드럽게 상기시킨다
- **Sub-agent (독립 검증)**: 작업 직후 독립 시각으로 점검한다

### 5. 확장 포인트
- `security-auditor`, `perf-reviewer`, `docs-checker` 등 역할별 서브를 추가 가능
- 각 서브는 **좁은 책임 + 읽기 전용 + 작업자보다 강하거나 같은 모델** 원칙을 동일하게 적용
- 일반 환경으로 복귀 시 역할을 한 칸씩 올리기: 작업자 Sonnet → 검증자 Opus
